As you can see, it's a Django app with it's own views, models, tests,
migrations, forms, etc.

** Goal

What we want to do is make Shiso's admin feature a /module/ in the same way
Django's admin is an /app/.

** Critique of Django templating system

In Django, the templating is set up so that templates can be overridden.
Overriding templates is accomplished by adding an admin folder to the
project-level templates folder and making template files with the same name as
the template files you want to override.

Overriding works for any installed Django app. If you install a third-party app
like Markdownx, you can customize the templates by overriding them.

I believe this system is necessary partially because Django is designed to allow
users to extend or modify app behavior without touching the original code. This
follows a philosophy of maximizing extensibility through OOP abstractions.

However, the desire to extend dependencies without touching them follows from a
general philosophical or aesthetic comfort (or even desire) to layer
abstractions and dependencies on top of each other, thereby creating more
feature-rich applications with little effort.

A preference for /abstraction layering/ over /abstraction flattening/ is, I
believe, a weakness of the Django philosophy and design in general.

** Our Alternative

Instead of implementing extra layers of abstraction to enable extensibility, my
proposal is to /vendor/ Shiso modules, whether they are first-party Shiso
modules or third-party modules/extensions, and thereby /flatten/ abstractions
and remove indirection. This follows the general Almighty philosophy of
preferring simplicity over complexity, direct over indirect approaches.
Vendoring dependencies and thereby adopting third-party code as you own, with
the intention of reducing dependencies over time, is the logical result of a
preference for simplicity.

I believe vendoring simplifies the process of customizing templates/forms, etc.
because there is no indirection through the override mechanism. If you want to
customize Shiso's admin components or forms, you simply go into the admin module
directory and make your modifications. If you want to do large customizations of
both templating and features, you can copy the entire admin module, rename it,
and add it to your =define-application=. After that, you can begin the work of
making a custom admin.

** Critique of this alternative

First, I would like you to provide a thoughtful critique of my own view, along
with a defense of Django's design decisions. I agree with Django on making
"admin" a module/app equal to any other Shiso module, but instead of
partitioning it to some .venv directory far away from the user, I propose
bringing all dependency code into the user's project. Is there some hidden
complexity in this design decision I haven't considered? What are the
trade-offs? Is it the decision actually consistent with my philosophy?
